Add a committed developer helper for starting local port-forwards to the Grafana, OpenTelemetry Collector, Loki, and Prometheus services so local server telemetry has an explicit path into the observability stack.

## Implementation Notes

Added `deploy/observability/local/port-forward.sh` as the preferred local development helper for observability port-forwards. The script forwards:

- Grafana `3000:3000`
- Prometheus `9090:9090`
- Loki `3100:3100`
- Tempo `3200:3200`
- OpenTelemetry Collector OTLP gRPC `4317:4317`
- OpenTelemetry Collector OTLP HTTP `4318:4318`

The helper runs in the foreground, accepts an optional namespace argument that defaults to `royale-observability`, prints local URLs and the server OTLP environment variable, and cleans up all child `kubectl port-forward` processes on interrupt.

Updated `deploy/observability/local/README.md` and `architecture/observability` to document the helper and clarify that port-forwards remain local developer processes rather than Kubernetes deployment behavior.

## Validation

- `bash -n deploy/observability/local/port-forward.sh`
- Sandboxed helper run was blocked from contacting the local Kubernetes API at `127.0.0.1:26443`, matching existing sandbox behavior.
- Elevated `deploy/observability/local/port-forward.sh` started all six port-forwards successfully.
- Forwarded health checks returned HTTP 200 for Grafana `/api/health`, Prometheus `/-/healthy`, Loki `/ready`, and Tempo `/ready`.
