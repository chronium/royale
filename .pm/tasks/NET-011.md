Connect executable client and dedicated server networking end to end: client handshake, redundant input send, server input receive, server snapshot send, client snapshot receive, and simple visual remote-player placeholders so two connected clients can see each other before interpolation, prediction, and reconciliation are refined.

## Acceptance Criteria

- The dedicated server executable can listen for real UDP clients, accept handshakes, allocate authoritative players, receive `ClientInput` packets, enqueue validated commands into the authoritative simulation, and send recipient-specific `ServerSnapshot` packets.
- The client executable can use `--connect` and `--port` to connect to a dedicated server, complete the handshake, send gameplay-facing input through `ClientInputSender`, receive and decode `ServerSnapshot` packets, and maintain a minimal client-side network state.
- Two local clients connected to one server can see both players represented in the world. Remote players may use simple placeholder geometry, capsules, or debug-style meshes; polished animation is not required.
- The local client should still be operable enough to move and cause authoritative snapshot changes, even if local prediction and reconciliation are not yet implemented.
- The SDL client must not reference `Royale.Server`; the server remains free of SDL GPU, ImGui, rendering, and client UI dependencies.

## Non-Goals

- Smooth interpolation is deferred to `NET-006`.
- Local movement prediction is deferred to `NET-007`.
- Reconciliation is deferred to `NET-008`.
- Bad-network simulation is deferred to `NET-009`.
- Cross-platform Mac-to-Linux validation remains deferred to `NET-010`.
