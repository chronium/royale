Expose player-specific debug information such as position, health, ammunition, connection identity, and last processed input through structured logs or an explicit debug endpoint instead of high-cardinality Prometheus labels.

## Implementation Notes

Implemented with structured server logs only. No debug HTTP endpoint was added, no protocol snapshot fields were added, and no player-specific Prometheus metric labels were introduced.

`Royale.Server` now exposes a read-only `ServerPlayerDebugState` projection through the in-process session and network runtime. The projection includes server tick, optional peer ID, connection ID, player ID, position, velocity, yaw, pitch, health, max health, alive state, weapon ID, magazine ammo, reserve ammo, reload state, last processed input sequence, and queued input count.

`ServerObservability` emits one structured `Authoritative player debug state` log event per player at a bounded cadence. The default interval is 60 ticks, matching one event per second at the current 60 Hz server tick rate. Empty-player runtimes emit no player debug log events.

The Wattle scenario test surface now exposes `scenario.observe.debugPlayer(player)` for the read-only debug projection. Added `in-process-player-debug-state.wattle` to keep connect, queued-input, and processed-input debug state covered as a gameplay integration scenario.

Updated `architecture/observability` with the per-player debug log behavior and the explicit logs-not-metrics cardinality rule.

PM hygiene: `NET-009` now depends on `NET-007` so bad-network validation waits for local movement prediction.
