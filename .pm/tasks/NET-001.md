Use LiteNetLib as the real UDP transport, pinned through BUILD-017, and hide it behind a small game-owned transport abstraction. LiteNetLib types must not leak into protocol, simulation, or gameplay code; the abstraction should support later in-process and simulated-network transports with the same message flow.

## Implementation Notes

- Added `Royale.Network` with game-owned transport types and `LiteNetLibNetworkTransport`.
- `Royale.Network` is the only production project that references `thirdparty/repos/LiteNetLib/LiteNetLib/LiteNetLib.csproj`.
- Transport payloads are opaque byte buffers; no protocol framing, game handshake, serialization, input messages, snapshots, prediction, reconciliation, or client/server launch wiring were added.
- Incoming LiteNetLib connection requests are accepted at the transport layer for this task. Protocol compatibility checks remain deferred to later connection/protocol work.
- `LiteNetLibNetworkTransport` configures LiteNetLib channels `0` through `63`; `Send` rejects larger channel values explicitly.
- Updated `architecture/networking` to document the `Royale.Network` boundary, LiteNetLib wrapper ownership, and deferred connection/protocol work.
