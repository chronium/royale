Connect executable client and dedicated server networking end to end: client handshake, redundant input send, server input receive, server snapshot send, client snapshot receive, and simple visual remote-player placeholders so two connected clients can see each other before interpolation, prediction, and reconciliation are refined.

## Acceptance Criteria

- The dedicated server executable can listen for real UDP clients, accept handshakes, allocate authoritative players, receive `ClientInput` packets, enqueue validated commands into the authoritative simulation, and send recipient-specific `ServerSnapshot` packets.
- The client executable can use `--connect` and `--port` to connect to a dedicated server, complete the handshake, send gameplay-facing input through `ClientInputSender`, receive and decode `ServerSnapshot` packets, and maintain a minimal client-side network state.
- Two local clients connected to one server can see both players represented in the world. Remote players may use simple placeholder geometry, capsules, or debug-style meshes; polished animation is not required.
- The local client should still be operable enough to move and cause authoritative snapshot changes, even if local prediction and reconciliation are not yet implemented.
- The SDL client must not reference `Royale.Server`; the server remains free of SDL GPU, ImGui, rendering, and client UI dependencies.

## Non-Goals

- Smooth interpolation is deferred to `NET-006`.
- Local movement prediction is deferred to `NET-007`.
- Reconciliation is deferred to `NET-008`.
- Bad-network simulation is deferred to `NET-009`.
- Cross-platform Mac-to-Linux validation remains deferred to `NET-010`.

## Implementation Notes

- Added `NetworkServerRuntime` in `Royale.Server`. It owns UDP polling, handshake acceptance, peer-to-authoritative-player mapping, server input queueing, fixed authoritative session stepping, due snapshot sending, and disconnect cleanup.
- Added `NetworkClientRuntime` in `Royale.Client`. It starts client UDP from an ephemeral local port, delays `NetworkHandshakeClient` creation until the transport `Connected` event, creates `ClientInputSender` after `ServerAccept`, sends fixed-tick input commands with immediate local look, and stores the latest decoded `ServerSnapshot`.
- Connect mode no longer creates or steps `LocalPlayerController`. Offline mode is unchanged. Connect-mode camera uses authoritative local snapshot position plus immediate local look, with an origin fallback before the first local snapshot.
- Snapshot players render as debug capsules with distinct local and remote colors through `DebugSceneBuilder`; this is the intended first validation surface for F6/F7. Polished solid player rendering, interpolation, prediction, reconciliation, and bad-network controls remain deferred to their existing follow-up tasks.
- Dependency boundaries are covered by tests: `Royale.Client` does not reference `Royale.Server`, and `Royale.Server` does not reference SDL, ImGui, or client rendering/UI projects.
- Fixed the human-validation movement bug by sending `ClientInput` with `NetworkDelivery.Sequenced` on channel `2`. LiteNetLib plain `Unreliable` packets do not preserve the requested channel, so the server previously received input packets on channel `0` and ignored them.
- Added `NetworkRuntimeLoopbackTests.ClientInputMovesAuthoritativePlayerOverUdpRuntime` to prove a real UDP client input packet is acknowledged and moves the authoritative player.
- Updated `architecture/networking` and `architecture/runtime-processes` to describe the executable UDP runtime flow, input delivery mode, and deferred work.
