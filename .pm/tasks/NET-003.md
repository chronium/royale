Validate protocol compatibility, accept clients, assign player identities, and transmit initial state.

## Implementation Notes

- Added `Royale.Protocol` handshake DTOs and binary payload serialization for `ClientHello`, `ServerAccept`, and `ServerReject`.
- Added deterministic development compatibility constants: `BuildId = "dev-build"` and `ContentVersion = "dev-content-1"`.
- Added fixed handshake string limits for build id, content version, map id, and reject detail.
- Added stable `ServerRejectReason` wire values for malformed packets, unexpected message types, unsupported protocol versions, incompatible build/content, and accept failures.
- Added `Royale.Network` handshake client/server handlers on top of `INetworkTransport` and `ProtocolPacketFramer`.
- `ClientHello` is sent with `SessionId = 0`; `ServerAccept` is sent with a nonzero server-assigned session id; pre-session `ServerReject` is sent with `SessionId = 0`.
- Handshake packets use reliable ordered delivery on channel `0`.
- The client exposes `Pending`, `Accepted`, `Rejected`, and `Disconnected` states and stores accepted or rejected payload details.
- The server accepts compatible hellos through a callback returning `NetworkHandshakeAcceptResult`, rejects incompatible or malformed pre-session packets where possible, and resends the existing accept payload for duplicate hellos from an already accepted peer.
- Added an integration test showing the accept callback can be backed by `InProcessServerSession.ConnectClient()` and still enqueue the existing initial snapshot.

## Deferred Scope

- Player input packet serialization is still deferred.
- Full `ServerSnapshot` serialization and snapshot streaming are still deferred.
- Client prediction, reconciliation, interpolation, and real executable client/server launch wiring are still deferred.
