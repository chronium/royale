Consume at most one queued client input command per connected client per server simulation tick so short press/release bursts are preserved without changing snapshot cadence. Add debug visibility for the processed command client tick and document that input hold/grace remains deferred.

## Acceptance Notes

- `InProcessServerSession.Step()` now dequeues at most one command per connected client per server simulation tick and leaves remaining commands queued for later ticks.
- Commands are consumed oldest-first per client, so press/release bursts can produce one authoritative movement tick before the release is processed.
- Snapshot cadence remains unchanged at 20 Hz via `ServerSnapshotSender` (`serverTick % 3 == 0`).
- No server-side input hold/grace was added.
- The authoritative player state now records `LastProcessedInputClientTick` alongside `LastProcessedInputSequence`; `ServerPlayerDebugState` and player debug structured logs expose both values.
- Updated `architecture/networking` to document simulation-tick-order command consumption, unchanged snapshot cadence, and deferred input hold/grace.
